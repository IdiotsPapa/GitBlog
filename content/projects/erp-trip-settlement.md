## 한 줄 요약

사내 ERP의 **출장정산** 영역에서 신청·정산·리포트·기준정보 화면을 구성·개발·운영했습니다.  
사내 웹 시스템을 **기능 개발부터 유지보수**까지 담당한 사례입니다.  
(회사·고객 실명 및 내부 화면·식별자는 보안상 공개하지 않습니다.)

## 배경

조직 내 출장·교육 비용이 신청·결재·정산·조회로 이어지는 **사내 업무 시스템**이 필요했습니다.  
ERP·전자결재와 맞물린 흐름 위에서, 현업이 매일 쓰는 화면을 안정적으로 만드는 것이 목표였습니다.

## 역할

- 출장정산 관련 **화면 구성 및 기능 개발**
- 신청 → 정산 → 조회(Report) → 기준정보 설정까지 **모듈 단위 구현**

## 구성한 기능

| 영역 | 내용 |
|------|------|
| 출장 신청 / 정산 | 개별·동행 출장 신청, 정산자 기준 비용 정산, 동행자 목록 연동 |
| 교육 신청 / 정산 | 교육도 동일하게 개별·동행(참석자) 그룹 처리 후 정산 |
| Report | 신청·정산 현황 조회·리포트 |
| 기준정보 설정 | 정산 한도·마스터 등 기준 데이터 관리 화면 |

## 기술 스택

- C# / **ASP.NET MVC4 → Core 3.1 → .NET 8**
- MSSQL (신청·정산·동행 그룹 연계 조회)
- 결재 연동(출장 신청/정산, 교육 신청/정산 등 유형별)
- 사내 ERP 공통 프레임·권한·메뉴 체계에 맞춘 화면 개발

## 개별 vs 동행 — 설계 포인트

출장은 **혼자 가는 경우**와 **여러 명이 함께 가는 경우**가 한 프로세스에 섞입니다.  
신청 인원 수와 정산 주체를 기준으로 경우의 수를 나눠 설계·구현했습니다.

| 구분 | 처리 |
|------|------|
| 개별 (1명) | 출장자 1건으로 신청·정산. 동행 그룹 번호 없음 |
| 동행 (2명+) | **그룹 번호**를 발급해 같은 출장으로 묶고, **인원별 신청 행**을 각각 저장 |
| 인원별 속성 | 차량 운행 여부·유류비 경로 등은 **사람마다 다를 수 있어** 인원 단위로 저장 |
| 첨부·결재 | 동행 시 첨부·결재 흐름을 인원/대표 기준으로 맞춘 뒤, 필요 시 동행 인원에 맞춤 |
| 정산 화면 | **정산자** 기준으로 상세·요약을 작성하고, 같은 그룹의 **동행 출장자 목록**을 함께 표시 |
| 교육 | 교육 신청/정산도 **동행(참석자) 목록 + 그룹 번호** 패턴을 동일하게 적용 |

즉 “한 건의 출장”과 “여러 명의 정산 주체”를 분리해,  
동행이어도 **누가 얼마를 정산하는지**가 섞이지 않게 만들었습니다.

## 작업에서 신경 쓴 점

- **신청과 정산**을 같은 업무 흐름으로 묶어, 사용자가 단계를 헷갈리지 않게 화면을 나눔
- 개별/동행 경우의 수를 **그룹 키 + 인원별 데이터**로 풀어, 신청·정산·조회가 일관되게 동작하도록 구현
- 출장·교육처럼 **유사하지만 기준이 다른 업무**를 공통 패턴(그룹·동행 목록·정산자)과 개별 기준으로 분리
- Report·기준정보·정산 한도를 함께 두어, 단순 CRUD가 아니라 **운영 가능한 모듈**로 구성

## 비고

- 상세 화면 캡처, 내부 URL, DB·계정 정보는 포함하지 않습니다.
- 이력서·면접에서는 담당 모듈·기술 스택·의사결정 위주로 설명합니다.
