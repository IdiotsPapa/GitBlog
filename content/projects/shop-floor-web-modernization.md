## 프로젝트 개요

생산현장 작업자가 작업지시·생산 진행·실적·비가동·불량 등을 처리하는 단말 시스템의 유지보수와 웹 전환 사례입니다.

현업 프로젝트이므로 회사·고객·공장·장비·데이터베이스를 식별할 수 있는 정보와 실제 화면은 공개하지 않습니다.

- 기존 설치형 시스템 유지보수: 2019년 ~ 2023년 확인 가능
- 웹 전환 PoC: 2026년 7월
- 역할: 레거시 분석·유지보수, 웹 전환 구조 및 일부 기능 개발

## 기존 시스템

기존 시스템은 현장 PC에 설치되는 Windows 데스크톱 애플리케이션이었습니다.

- 작업장·라인·근무조 선택
- 작업지시와 계획수량 확인
- 작업 시작·종료
- 생산실적·불량·재투입 처리
- 설비 비가동 등록·해제
- 바코드·포장·라벨 등 현장 보조 기능
- 현장별 예외 처리와 외부 시스템 연계

오랜 기간 운영된 만큼 기능은 많았지만, 오래된 UI 라이브러리와 플랫폼 의존성, 설치·업데이트 부담이 있었습니다.

## 웹 전환 방향

```text
기존 설치형 C# 애플리케이션
         ↓ 업무 규칙 분석
.NET 8 REST API + React/TypeScript 키오스크 UI
```

기존 프로그램을 한 번에 교체하기보다, 조회와 공통 기능부터 분리해 브라우저 기반 화면으로 옮기는 **점진적 전환**을 선택했습니다.

## 현재 구현 범위

- .NET 8 기반 REST API 골격
- React / TypeScript / Vite 기반 키오스크 UI
- 서버 연결 상태 확인
- 작업장 선택
- 라인 상태 조회
- 로컬 개발용 데이터 재현 환경

인증, 전체 업무 화면, 현장 장비 연동과 종단 간 검증은 아직 완료되지 않은 PoC 단계입니다.

## 기술 스택

### 기존

- C# / .NET Framework / WinForms
- 데스크톱 UI 컴포넌트
- 관계형 데이터베이스
- 바코드·라벨·현장 장비 연동

### 전환

- .NET 8 / REST API
- React / TypeScript / Vite
- 브라우저 키오스크 UI

## 전환하며 중요하게 본 점

### 화면보다 업무 규칙이 먼저다

설치형 화면을 그대로 웹으로 복사하면 기술만 바뀌고 복잡성은 남습니다. 작업 시작·마감·비가동처럼 상태 전이가 있는 업무 규칙과 현장별 예외를 먼저 분리해야 했습니다.

### 현장은 네트워크가 항상 안정적이지 않다

브라우저 화면은 서버 연결에 더 의존할 수 있습니다. 연결 상태 표시, 재시도, 중복 요청 방지와 로컬 상태·서버 상태의 일관성을 고려해야 합니다.

### 장비 연동은 별도 전략이 필요하다

바코드·라벨·각인처럼 로컬 장비를 사용하는 기능은 일반 웹 API만으로 옮기기 어렵습니다. 브라우저 지원 범위, 로컬 브리지, 별도 에이전트 중 어떤 방식을 쓸지 기능별로 판단해야 합니다.

### 작은 범위부터 검증한다

장기간 운영된 시스템은 숨은 예외가 많습니다. 조회·공통 기능부터 옮기고 기존 시스템과 결과를 비교하면서 전환해야 회귀 위험을 줄일 수 있습니다.

## 현재 상태

기존 설치형 시스템은 장기간 유지보수된 운영 시스템이고, 웹 전환본은 일부 조회와 화면 구조를 검증한 **초기 PoC**입니다.

이 프로젝트는 최신 기술 적용 자체보다, **레거시 업무를 이해하고 운영을 멈추지 않은 채 점진적으로 고도화하는 과정**을 보여 주는 사례입니다.
