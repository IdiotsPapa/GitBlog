여러 서버와 데이터 연계 작업의 상태를 한곳에서 확인하기 위해 만든 사내 운영 지원 시스템입니다.

- 작업 기간: 2021년 5월 ~ 2022년 8월 확인 가능
- 상태: 개발 후 운영·유지보수
- 기술: C# / .NET Core · ASP.NET Core, Windows Agent, Vue, Oracle · MSSQL

당시에는 서버마다 직접 접속하거나 담당자가 따로 상태를 확인해야 했습니다. 서비스가 멈춘 뒤 장애를 발견하기보다, 자원 사용량·연계 적체 같은 신호를 먼저 알아채도록 서버별 에이전트가 상태를 모으고 중앙 웹에서 조회·알림하는 구조를 만들었습니다.

```text
서버별 Agent
  → 상태·운영 지표 수집
  → 중앙 API 및 저장소
  → 웹 대시보드
  → 장애·복구 알림
```

## 직접 만든 기능

CPU·메모리·디스크·프로세스·Windows 서비스 상태를 수집하고 임계치를 넘으면 경보를 보냅니다. DB 연결·응답 시간과 장기 실행 작업·적체량도 함께 확인합니다. 관리자 화면에서는 서버·점검 항목·담당자·알림 정책을 관리하고, 기간별 상태와 응답 시간을 차트·목록으로 조회합니다.

장애·복구 시 이메일·문자로 알리고, 제한된 조건에서는 프로세스·서비스를 재시작할 수 있게 했습니다.

## 장기 실행 에이전트의 연결 자원 개선

상시 실행 에이전트는 주기적으로 여러 Oracle·MSSQL의 연결·응답을 검사합니다. 장기 실행 서비스에서는 연결을 닫아도 공급자 연결 풀이 물리 연결과 내부 자원을 남길 수 있어, 시간이 지날수록 메모리·연결 자원이 쌓일 가능성을 점검했습니다.

공통 연결 코드에 `Pooling=false`를 적용하고, 쿼리·프로시저용 연결·명령은 `using` 범위에서 정리했습니다. 이 내용은 에이전트 **v1.0.8** 변경 기록에 남아 있습니다. 전후 메모리 수치는 없어 감소율은 제시하지 않으며, 상시 실행 서비스에서는 외부 연결 수명까지 관리해야 한다는 점을 확인한 사례로 정리합니다.

## 운영하면서 배운 점

모니터링은 데이터를 많이 모으는 것보다 필요한 신호를 놓치지 않는 쪽이 중요했습니다. 장애·복구를 구분해 중복·오탐 알림을 줄여야 했고, 서비스 재시작은 조건·이력을 제한해 안전장치를 뒀습니다. 에이전트 자체도 운영 대상이라 타이머 중복·비동기 예외·연결 수명·권한을 함께 살펴야 했습니다.

내부 서버명·IP·접속 정보·데이터 구조·식별 정보는 보안상 공개하지 않습니다.
