## 프로젝트 개요

여러 서버와 데이터 연계 작업의 상태를 한곳에서 확인하기 위한 **사내 운영 지원 시스템**입니다.

각 서버에 설치되는 백그라운드 에이전트가 상태를 수집하고, 중앙 웹 시스템이 이를 저장·시각화하는 구조로 구성했습니다. 회사·서버·인프라를 식별할 수 있는 정보는 보안을 위해 공개하지 않습니다.

- 작업 기간: 2021년 5월 ~ 2022년 8월 확인 가능
- 형태: 에이전트 + 중앙 API + 관리자 웹
- 역할: 기능 개발 및 운영·유지보수

## 해결하려던 문제

서버 장애는 서비스가 완전히 멈춘 뒤 발견하는 것보다, 자원 사용량 증가나 연계 적체 같은 **선행 신호**를 일찍 알아채는 것이 중요합니다.

기존에는 서버마다 직접 접속하거나 담당자가 별도로 상태를 확인해야 했습니다. 이를 줄이기 위해 다음 흐름을 만들었습니다.

```text
서버별 Agent
  → 상태·운영 지표 수집
  → 중앙 API 및 저장소
  → 웹 대시보드
  → 장애·복구 알림
```

## 구현 범위

- CPU·메모리·디스크 사용량 수집과 임계치 경보
- 프로세스 및 Windows 서비스 상태 확인
- 데이터베이스 연결 상태와 응답 시간 점검
- 장기 실행 작업·적체량 등 운영 지표 조회
- 이메일·문자 기반 장애 및 복구 알림
- 서버·점검 항목·담당자·알림 정책 관리
- 기간별 상태와 응답 시간을 확인하는 차트·목록
- 제한된 조건에서 프로세스·서비스 재시작

## 기술 구성

- C# / .NET Core
- ASP.NET Core 기반 중앙 웹·API
- Windows 백그라운드 서비스
- Vue 기반 관리자 화면
- 관계형 데이터베이스
- Windows 시스템 관리 API

## 개발하며 중요하게 본 점

### 경보는 많다고 좋은 것이 아니다

같은 장애가 반복 감지될 때 알림이 계속 쌓이면 담당자는 오히려 중요한 신호를 놓칩니다. 장애와 복구 상태를 구분하고, 중복·오탐을 줄이는 것이 단순 수집보다 중요했습니다.

### 자동 복구에는 안전장치가 필요하다

서비스 재시작은 편리하지만 잘못 실행하면 장애를 키울 수 있습니다. 실행 조건, 대상 제한, 이력 기록처럼 **오작동을 막는 제약**이 필요했습니다.

### 모니터링 자체도 운영 대상이다

타이머 중복 실행, 비동기 예외, 연결 수명, 높은 Windows 권한 등 모니터링 시스템 자체의 안정성도 함께 관리해야 했습니다.

## 돌아보며

이 프로젝트를 통해 운영·유지보수는 단순히 장애가 발생했을 때 고치는 일이 아니라, **문제를 빨리 발견할 수 있는 구조를 만드는 일**이라는 점을 배웠습니다.

현재 기준으로는 지원이 종료된 런타임과 오래된 프론트엔드 의존성 등 기술 부채가 있습니다. 다시 고도화한다면 최신 .NET LTS 전환, 최소 권한 적용, 구조화 로그, 경보 중복 억제 정책부터 정리할 계획입니다.

## 공개 범위

내부 서버명, IP, 접속 정보, 데이터 구조, 회사·고객 식별 정보와 실제 화면은 포함하지 않았습니다.
