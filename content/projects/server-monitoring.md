여러 서버와 데이터 연계 작업의 상태를 한곳에서 확인하기 위해 만든 사내 운영 지원 시스템입니다.

- 작업 기간: 2021년 5월 ~ 2022년 8월 확인 가능
- 상태: 기능 개발 완료 후 운영·유지보수

당시에는 서버마다 직접 접속하거나 담당자가 따로 상태를 확인해야 했습니다. 서비스가 완전히 멈춘 뒤 장애를 발견하기보다 자원 사용량 증가나 연계 적체 같은 신호를 먼저 알아챌 수 있도록, 각 서버의 백그라운드 에이전트가 상태를 모으고 중앙 웹 시스템에서 저장·조회하는 구조를 만들었습니다.

```text
서버별 Agent
  → 상태·운영 지표 수집
  → 중앙 API 및 저장소
  → 웹 대시보드
  → 장애·복구 알림
```

## 직접 만든 기능

CPU·메모리·디스크 사용량과 프로세스, Windows 서비스 상태를 수집하고 임계치를 넘으면 경보를 보내도록 했습니다. 데이터베이스 연결 여부와 응답 시간, 장기 실행 작업과 적체량 같은 운영 지표도 함께 확인할 수 있었습니다. 관리자 화면에서는 서버와 점검 항목, 담당자, 알림 정책을 관리하고 기간별 상태와 응답 시간을 차트와 목록으로 조회했습니다.

장애가 발생하거나 복구되면 이메일과 문자로 알렸고, 제한된 조건에서는 프로세스와 서비스를 재시작할 수 있게 했습니다. 구현에는 C#/.NET Core, ASP.NET Core 기반 웹·API, Windows 백그라운드 서비스, Vue 관리자 화면, 관계형 데이터베이스와 Windows 시스템 관리 API를 사용했습니다.

## 운영하면서 배운 점

모니터링은 데이터를 많이 모으는 것보다 필요한 신호를 놓치지 않는 쪽이 더 중요했습니다. 같은 장애를 계속 감지할 때마다 알림을 보내면 담당자가 금방 지치기 때문에 장애와 복구 상태를 구분하고 중복과 오탐을 줄여야 했습니다.

서비스 재시작도 무조건 자동화할 수 있는 기능은 아니었습니다. 잘못 실행하면 오히려 장애를 키울 수 있어 실행 조건과 대상을 제한하고 이력을 남기는 안전장치가 필요했습니다.

모니터링 프로그램 자체도 운영 대상이었습니다. 타이머 중복 실행, 비동기 예외, 연결 수명, 높은 Windows 권한처럼 수집 기능 바깥의 문제도 꾸준히 살펴야 했습니다. 이 경험을 통해 운영·유지보수는 장애가 난 뒤 고치는 일뿐 아니라, 문제를 일찍 발견할 수 있는 구조를 만드는 일이라는 점을 배웠습니다.

현재 기준으로는 지원이 끝난 런타임과 오래된 프론트엔드 의존성이 남아 있습니다. 다시 손본다면 최신 .NET LTS 전환, 최소 권한 적용, 구조화 로그와 경보 중복 억제 정책을 먼저 정리할 생각입니다.

내부 서버명과 IP, 접속 정보, 데이터 구조, 회사·고객 식별 정보 및 실제 화면은 보안을 위해 공개하지 않습니다.
