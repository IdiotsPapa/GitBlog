## 프로젝트 개요

해외 쇼핑몰의 상품 콘텐츠를 복제하지 않고, 공개 랭킹의 순위·가격·리뷰 수·평점 같은 최소 신호만으로 변화 흐름을 분석하려 한 프로토타입입니다.

- 집중 구현: 2026년 1월 30일
- 상태: 모듈별 코드가 있는 통합 프로토타입, 운영·배포 미검증

## 설계 원칙

이 프로젝트에서 가장 먼저 정한 것은 “무엇을 수집할지”보다 **무엇을 수집하지 않을지**였습니다.

- 상품 이미지 저장 금지
- 상품명·브랜드 저장 금지
- 원본 URL·구매 링크 저장 금지
- 동일 항목 비교에 필요한 식별자는 단방향 해시로 변환

데이터 최소화만으로 법적 안전성을 보장할 수는 없으며, 실제 운영 전에는 대상 사이트의 이용약관과 robots 정책을 별도로 검토해야 합니다.

## 목표 아키텍처

```text
랭킹 페이지
  → 크롤러
  → 식별자 해싱·정제
  → PostgreSQL 원시 스냅샷
  → 이전 시점과 변화량 비교
  → 트렌드 점수
  → FastAPI
  → React 대시보드
```

## 구현한 범위

- 랭킹 페이지 요청·파싱 골격
- 식별자 SHA-256 해싱과 원본 제거
- 순위 변화량·리뷰 증가량 분석기
- 순위·리뷰·평점 기반 규칙 점수
- SQLAlchemy 모델과 PostgreSQL DDL
- 원시 데이터 → 변화량 → 점수 파이프라인
- FastAPI 헬스 체크·상위 트렌드·이력 API
- React 요약 카드와 급상승 목록
- 점수 계산기 단위 테스트
- Ubuntu / systemd / cron / Nginx 배포 문서

## 기술 스택

- Python / FastAPI / SQLAlchemy
- PostgreSQL
- Requests / BeautifulSoup
- React / Vite / Tailwind CSS
- Axios / Recharts

## 기술적으로 고민한 점

### 최소 데이터로 동일 항목을 추적하기

원본 식별자를 저장하지 않으면서 시점별 변화를 비교하기 위해, 식별자를 단방향 해시로 변환해 같은 항목의 키로 사용했습니다.

### 서로 다른 지표를 하나의 점수로 만들기

순위, 리뷰 증가, 평점은 단위가 다릅니다. 각 지표를 제한된 범위로 정규화하고 가중치를 적용하는 규칙 기반 점수를 설계했습니다. 신규 진입처럼 이전 값이 없는 경우의 정책도 별도로 필요합니다.

### 재실행 가능한 파이프라인

같은 시점의 점수를 다시 계산할 때 중복 삽입하지 않도록 upsert와 트랜잭션 경계를 명확히 해야 합니다. 프로토타입에서는 이 부분이 충분하지 않아 후속 개선 항목으로 남았습니다.

## 검토하며 발견한 한계

- 실제 파서가 리뷰·평점·카테고리를 충분히 추출하지 못함
- robots 정책 확인이 코드로 구현되지 않음
- 변화량 생성 주기와 API 조회 주기가 일치하지 않는 부분
- 일부 import·모델·DDL 불일치
- 대시보드 차트 미연결
- API·크롤러·DB 통합 테스트와 실제 배포 근거 없음

따라서 “실시간 트렌드 서비스를 운영했다”가 아니라, **데이터 최소화 원칙을 반영한 수집·분석 구조를 설계하고 모듈화한 프로토타입**으로 기록합니다.
