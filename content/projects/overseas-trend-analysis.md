2026년 1월 30일에 집중해서 만든 트렌드 분석 프로토타입입니다. 해외 쇼핑몰의 상품 콘텐츠를 복제하는 대신, 공개 랭킹에서 순위·가격·리뷰 수·평점 같은 최소 신호만 가져와 시점별 변화를 계산해 보려 했습니다.

처음 정한 것은 수집 대상보다 수집하지 않을 데이터였습니다. 상품 이미지, 상품명과 브랜드, 원본 URL과 구매 링크는 저장하지 않고, 같은 항목을 비교하는 데 필요한 식별자는 SHA-256 단방향 해시로 바꾼 뒤 원본을 제거했습니다. 다만 데이터 최소화가 법적 안전성을 보장하는 것은 아닙니다. 실제 운영 전에는 대상 사이트의 이용약관과 robots 정책을 따로 검토해야 하며, robots 정책 확인은 코드에도 구현하지 못했습니다.

## 만들었던 흐름

랭킹 페이지를 요청하고 파싱하는 골격에서 시작해, 정제한 스냅샷을 PostgreSQL에 저장하고 이전 시점과의 순위 변화량·리뷰 증가량을 계산하는 흐름을 구성했습니다. 순위, 리뷰, 평점은 단위가 달라 제한된 범위로 정규화한 뒤 가중치를 적용하는 규칙 기반 점수로 묶었습니다. 신규 진입처럼 이전 값이 없는 경우의 정책은 더 정해야 합니다.

데이터 쪽에는 SQLAlchemy 모델과 PostgreSQL DDL, 원시 데이터에서 변화량과 점수로 이어지는 파이프라인 코드가 있습니다. API는 FastAPI로 헬스 체크, 상위 트렌드, 이력 조회를 만들었고, React 화면에는 요약 카드와 급상승 목록을 구성했습니다. 점수 계산기 단위 테스트와 Ubuntu, systemd, cron, Nginx를 기준으로 한 배포 문서도 작성했습니다.

사용한 기술은 Python, FastAPI, SQLAlchemy, PostgreSQL, Requests, BeautifulSoup과 React, Vite, Tailwind CSS, Axios, Recharts입니다.

## 코드를 다시 보며 확인한 빈틈

모듈별 코드는 있지만 전체를 연결한 상태는 검증하지 못했습니다. 실제 파서는 리뷰·평점·카테고리를 충분히 추출하지 못하고, 일부 import와 모델, DDL도 서로 맞지 않습니다. 변화량 생성 주기와 API 조회 주기가 어긋나는 부분이 있으며 대시보드 차트도 연결되지 않았습니다.

같은 시점의 점수를 다시 계산할 때 중복 삽입을 막으려면 upsert와 트랜잭션 경계를 명확히 해야 하는데, 프로토타입에서는 충분히 다루지 못했습니다. API·크롤러·DB 통합 테스트가 없고 실제 배포 근거도 없습니다.

그래서 이 결과를 운영된 실시간 트렌드 서비스라고 볼 수는 없습니다. **데이터 최소화 원칙을 넣어 수집·분석 모듈을 구성해 본, 통합 미검증 프로토타입**으로 남깁니다.
